Краткое ТЗ: учет всех поступлений на расчетный счет через T-API (T-Банк)
1. Проблема

Текущая логика учета оплат, завязанная на статусы платежей по выставленным счетам и/или корректное указание клиентом номера счета в назначении платежа, не обеспечивает полный и достоверный учет. Клиенты часто ошибаются в назначении, не указывают номер счета или указывают неверно. В результате поступления не сопоставляются с конкретными счетами и часть денег остается неучтенной или учитывается с задержкой.

2. Цель

Обеспечить получение и регистрацию в системе всех входящих поступлений на расчетный счет независимо от того, указал ли клиент номер счета и корректно ли он его указал. Сопоставление поступлений с выставленными счетами выполняется отдельной логикой, но само поступление должно быть зафиксировано всегда.

3. Источник данных

T-API T-Банка:

Выписка по счету: GET /openapi/api/v1/statement

Вебхук: “Операция по счету” (уведомления о новых операциях и изменениях операций)

4. Основы решения

Решение строится на двух механизмах, которые работают совместно:

Вебхук “Операция по счету” (push)

Нужен для оперативного получения новых операций по счету.

Система принимает входящие события и сохраняет их для дальнейшей обработки.

Обработка должна быть идемпотентной (возможны повторы событий).

Регулярная сверка через выписку (pull)

Нужна для гарантированного покрытия всех поступлений и устранения возможных пропусков/изменений операций.

По расписанию (минимум 1 раз в сутки) запрашивается выписка за период и выполняется синхронизация данных в базе (upsert без дублей).

5. Что считается “поступлением”

Из данных выписки и вебхука учитываются только входящие операции (credit/income). Исходящие (debit/outcome) не входят в область задачи.

6. Требования к хранению данных

Для каждого поступления требуется сохранять минимум:

идентификатор операции (если присутствует в данных)

счет (accountNumber)

дата операции

сумма и валюта

назначение платежа (если передается)

сведения о плательщике, включая ИНН (если передается)

статус операции (если передается)

Данные должны храниться так, чтобы повторные события или повторная синхронизация не создавали дубликаты.

7. Автоматическая попытка идентификации неразнесенной оплаты

Если поступление не удалось однозначно сопоставить с конкретным счетом по номеру счета/назначению платежа, выполняется дополнительная проверка:

Из поступления берется ИНН плательщика (если он доступен в данных операции).

По ИНН в базе находится клиент и список его открытых (неоплаченных) счетов.

Выполняются правила разнесения:

Если найден ровно один неоплаченный счет клиента, сумма которого точно равна сумме поступления, то этот счет считается оплаченным, и выполняется привязка поступления к счету.

Если у клиента несколько неоплаченных счетов и сумма поступления не совпала точно ни с одним, то не предпринимать действий, поступление остается неразнесенным.

Если сумма поступления совпала более чем с одним неоплаченным счетом (несколько счетов на одинаковую сумму), то не предпринимать действий, поступление остается неразнесенным.

8. Результат

## План реализации по новому ТЗ (контроль отправки счетов)

### 1. Актуализация модели данных и логов
1. Добавить SQL patch в `sql/` для таблицы логов неудачных отправок счетов (например, `finance_send_failures`):
   - `id`, `document_id`, `invoice_plan_id` (nullable), `channel` (`email|telegram|diadoc`), `error_code`, `error_message`, `attempt_no`, `created_at`.
2. Добавить индексы:
   - `(document_id, created_at)`,
   - `(invoice_plan_id, created_at)`,
   - `(channel, created_at)`.
3. Зафиксировать правило хранения:
   - пишем только неудачные отправки;
   - успешные отправки в эту таблицу не пишем;
   - события открытия email продолжаем хранить в `finance_email_open_events` (уже есть пиксель).

### 2. Нормализация бизнес-статуса “отправленности” счета
1. Ввести вычисляемый статус отправки для карточки счета:
   - `sent_ok_all_channels` (успех по всем включенным каналам),
   - `sent_partial_or_failed` (хотя бы один включенный канал не отправлен),
   - `sent_failed_all` (не отправлен ни по одному включенному каналу).
2. Определение “включенных каналов” брать из настроек отправки конкретного счета/плана (email/telegram/diadoc).
3. Уточнить правило по задолженностям:
   - `sent_failed_all` не попадает в задолженности;
   - `sent_partial_or_failed` и `sent_ok_all_channels` участвуют в задолженностях по текущей логике оплаты.

### 3. Backend: фиксация ошибок отправки по каналам
1. В местах отправки счетов по Email/Telegram/Diadoc:
   - при ошибке писать запись в `finance_send_failures`;
   - при повторной отправке писать новую попытку (`attempt_no`).
2. Неуспешные попытки должны сохранять читабельный текст ошибки + технический код.
3. Для успешной отправки:
   - не создавать запись в `finance_send_failures`;
   - сохранять/обновлять существующие события отправки как сейчас (если уже ведутся в `finance_send_events`).
4. Добавить сервис для получения “сводки по каналам” для карточки:
   - какие каналы обязательны для счета;
   - по каким каналам есть последняя ошибка;
   - итоговый вычисляемый статус отправки.

### 4. Backend API для UI “Статус”
1. Расширить payload колонки “Выставлено, ожидание оплаты”:
   - `send_state` (`ok_all|failed_partial|failed_all`),
   - `send_channels_required`,
   - `send_channels_failed`,
   - `send_channels_success`,
   - `default_selected_for_resend` (true для `failed_*`, false для остальных).
2. Добавить endpoint/действие “Отправить выбранные”:
   - вход: список `document_id`/`invoice_plan_id`;
   - поведение: повторная попытка отправки строго по включенным у счета каналам;
   - результат: детальный ответ по каждому счету и каналу.
3. Обновить массовую операцию так, чтобы она не ломала идемпотентность и корректно логировала ошибки.

### 5. Frontend: изменения в колонке “Выставлено, ожидание оплаты”
1. Добавить справа в колонке кнопку `Отправить выбранные`.
2. В каждой карточке:
   - чекбокс в правом верхнем углу (`Отправить повторно`);
   - по умолчанию отмечен только для карточек с неудачной отправкой.
3. Ввести третий визуальный тип карточек `Неудачная отправка`:
   - выводить выше остальных карточек в колонке;
   - серый фон;
   - бейджи каналов:
     - красный для неуспешного канала,
     - зеленый для успешного канала,
     - не показывать канал, который не включен у клиента/счета.
4. Обновить сортировку карточек в колонке:
   - сначала `Неудачная отправка`,
   - затем остальные по текущему порядку.

### 6. Учет в “Задолженностях”
1. Скорректировать backend-выборку задолженностей:
   - исключать счета со статусом `sent_failed_all`;
   - включать счета, где отправка прошла хотя бы по одному каналу.
2. Добавить регрессионную проверку, что уже оплаченные счета не попадают в задолженности независимо от отправки.

### 7. Триггеры уведомлений по сбоям отправки
1. Подключить новый event (по вашему названию в настройках), например:
   - `finance.invoice.send_failed`.
2. Точка вызова триггера:
   - при появлении нового счета с неудачной отправкой;
   - при массовой операции, если есть хотя бы один счет с неуспехом.
3. Полезная нагрузка события:
   - список счетов,
   - каналы с ошибками,
   - короткие ошибки,
   - ссылка на раздел “Статус”.

### 8. Миграции и обратная совместимость
1. Подготовить SQL patch без деструктивных изменений.
2. Добавить fallback в коде, если новая таблица еще не применена (чтобы API не падал до миграции).
3. Отдельно приложить блок SQL для ручного запуска на проде.

### 9. Тест-план приемки
1. Кейс “успех по всем каналам”: карточка не в серой группе, чекбокс снят, в задолженности участвует.
2. Кейс “частичный успех”: карточка в `Неудачная отправка`, красно-зеленые бейджи, в задолженности участвует.
3. Кейс “провал по всем каналам”: карточка в `Неудачная отправка`, чекбокс отмечен, в задолженности не участвует.
4. Повторная отправка выбранных:
   - неудачи пишутся в лог;
   - успех удаляет канал из “красных” у карточки.
5. Email open tracking:
   - пиксель создает событие открытия;
   - отсутствие открытия не влияет на статус отправки счета.
6. Уведомления:
   - триггер срабатывает при появлении одного/нескольких проблемных счетов;
   - payload содержит список проблемных каналов.

4. UI колонки “Выставлено, ожидание оплаты” (чекбоксы, сортировка, бейджи, кнопка).
5. Триггер уведомлений о неудачной отправке.
6. Интеграционные проверки и финальная приемка по тест-плану.
